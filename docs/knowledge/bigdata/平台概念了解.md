## HDFS
<img src="/images/bigdata/1691548631398-b9c7003d-4d23-4633-880b-bb39cd4a4762.png" alt="HDFS架构">

### 主要概念
1. **nameNode**
+ NameNode用于存储、生成文件系统的元数据。运行一个实例  
+ 元数据：DataNode存储数据的位置等详细信息
2. **dataNode**
+  DataNode用于存储实际的数据，将自己管理的数据块信息上报给NameNode  
3. **zookeeper**
+  利用zookeeper实现主备NameNode，以解决单点NameNode故障问题  

### HDFS的高可靠性（HA）
+ HDFS的高可靠性（HA）主要体现在利用zookeeper实现主备NameNode，以解决单点NameNode故障问题。
+ ZooKeeper主要用来存储HA下的状态文件，主备信息。ZK个数建议3个及以上且为奇数个。
+ NameNode主备模式，主提供服务，备同步主元数据并作为主的热备。
+ ZKFC(ZooKeeper Failover Controller)用于监控NameNode节点的主备状态。
+ JN(JournalNode 日志节点)用于存储Active NameNode生成的Editlog。Standby NameNode（备用节点）加载JN上Editlog，同步元数据。

### ZKFC控制NameNode主备仲裁
+ ZKFC作为一个精简的仲裁代理，其利用zookeeper的分布式锁功能，实现主备仲裁，再通过命令通道，控制NameNode的主备状态。ZKFC与NN部署在一起，两者个数相同。

### 元数据同步
+ 主NameNode对外提供服务。生成的Editlog同时写入本地和JN，同时更新主NameNode内存中的元数据。
+ 备NameNode监控到JN上Editlog变化时，加载Editlog进内存，生成新的与主NameNode一样的元数据。元数据同步完成。
+ 主备的FSImage仍保存在各自的磁盘中，不发生交互。FSImage是内存中元数据定时写到本地磁盘的副本，也叫元数据镜像。

### 元数据持久化
+ EditLog:记录用户的操作日志，用以在FSImage的基础上生成新的文件系统镜像。
+ FSImage:用以阶段性保存文件镜像。
+ FSImage.ckpt:在内存中对fsimage文件和EditLog文件合并（merge）后产生新的fsimage，写到磁盘上，这个过程叫checkpoint.。备用NameNode加载完fsimage和EditLog文件后，会将merge后的结果同时写到本地磁盘和NFS。此时磁盘上有一份原始的fsimage文件和一份新生成的checkpoint文件：fsimage.ckpt. 而后将fsimage.ckpt改名为fsimage（覆盖原有的fsimage）。
+ EditLog.new: NameNode每隔1小时或Editlog满64MB就触发合并,合并时,将数据传到Standby NameNode时,因数据读写不能同步进行,此时NameNode产生一个新的日志文件Editlog.new用来存放这段时间的操作日志。Standby NameNode合并成fsimage后回传给主NameNode替换掉原有fsimage,并将Editlog.new 命名为Editlog。

## MapReduce
 当你向MapReduce框架提交一个计算作业时，它会首先把计算作业拆分成若干个**Map任务**，然后分配到不同的节点上去执行，每一个Map任务处理输入数据中的一部分，当Map任务完成后，它会生成一些中间文件，这些中间文件将会作为**Reduce任务**的输入数据。Reduce任务的主要目标就是把前面若干个Map的输出汇总到一起并输出

在Hadoop集群中，Kerberos、OpenLDAP、Knox和Ranger可以协同工作，以提供全面的安全性和访问控制。以下是一个具体的例子来说明它们在Hadoop中的协调：

**场景：**

假设有一个大型的金融机构，他们在Hadoop集群中存储大量的敏感客户数据和交易数据。他们需要确保数据安全，并只允许授权的用户和服务访问数据。同时，他们也希望外部的分析应用程序能够通过安全渠道访问集群的一部分数据。

**协调过程：**

1. **Kerberos和OpenLDAP协同工作：**
    - 首先，金融机构将使用OpenLDAP作为用户数据库，其中存储了员工的用户名、密码哈希等信息。OpenLDAP允许集中管理用户身份和组织结构。
    - 在Kerberos中，金融机构将配置KDC（Kerberos Key Distribution Center），KDC将与OpenLDAP集成以验证用户凭据。当用户登录时，Kerberos将向OpenLDAP发送身份验证请求并检查用户凭据的有效性。
    - 一旦用户通过Kerberos成功地登录，他们将获得TGT（Ticket-granting ticket），这将允许他们在Hadoop集群中的不同服务之间进行单点登录。
2. **Knox和Ranger协同工作：**
    - 金融机构将使用Ranger来设置和管理Hadoop集群内的细粒度访问控制策略。他们可以为不同的部门、角色或用户设置不同的权限，限制对特定数据和服务的访问。
    - 为了让外部的分析应用程序可以安全地访问集群中的一部分数据，金融机构将使用Knox作为安全访问网关。
    - 在Knox中，管理员将配置安全策略，以控制外部应用程序对Hadoop集群的访问。例如，他们可以指定只允许特定IP地址的应用程序访问特定的服务，或者要求外部应用程序通过特定的认证机制进行身份验证。
    - 外部应用程序通过Knox与Hadoop集群中的服务进行交互，Knox会处理来自应用程序的请求，并根据Ranger中设置的策略来确定是否允许访问。

**具体例子：**

1. 内部用户访问数据：
    - 内部员工登录Hadoop集群时，他们需要通过Kerberos进行身份验证。Kerberos与OpenLDAP集成，验证员工的用户名和密码哈希。
    - 一旦身份验证成功，Kerberos颁发TGT，允许员工在整个Hadoop集群中进行单点登录。
    - Ranger设置了细粒度的访问控制策略，根据员工所属部门或角色，限制他们对敏感客户数据的访问权限(ranger使用openLDAP的存储信息)。
2. 外部应用程序访问数据：
    - 外部分析应用程序需要访问集群中的一部分数据，但不能直接访问集群。
    - 金融机构使用Knox作为安全访问网关，配置了外部应用程序的安全策略。
    - Knox验证外部应用程序的身份，并根据Ranger中设置的策略来确定是否允许应用程序访问特定的服务或数据。

通过Kerberos、OpenLDAP、Knox和Ranger的协同工作，金融机构可以实现内部用户和外部应用程序的安全访问控制，保护敏感数据，提高集群的安全性和合规性。

LDAP和Ranger在Hadoop中扮演不同的角色，但它们之间可以结合使用，共同构建全面的用户权限管理和安全控制体系。

1. **LDAP（轻量级目录访问协议）：** LDAP是一种用于访问和维护分布式目录信息服务的协议，通常用于用户和组织的身份验证和信息存储。在Hadoop环境中，LDAP主要用于用户身份验证和用户信息管理。
    - 用户身份验证：Hadoop服务可以通过LDAP验证用户的身份，确保只有有效的用户可以访问Hadoop集群。
    - 用户信息管理：LDAP允许将用户信息和组织信息集中存储在一个目录中，方便管理员对用户进行管理和维护。
2. **Ranger（Apache Ranger）：** Ranger是一个集中化的授权和访问控制框架，专门用于提供Hadoop服务中的访问控制和安全策略管理。Ranger可以管理多个Hadoop组件，如HDFS、YARN、Hive、HBase等，并通过定义策略来控制用户对资源的访问权限。
    - 访问控制：Ranger允许管理员定义细粒度的策略，规定用户和组在Hadoop资源上可以执行的操作，从而实现灵活的访问控制。
    - 审计和监控：Ranger提供审计和监控功能，记录用户的操作和访问记录，有助于跟踪用户行为和检测潜在的安全问题。
3. **LDAP与Ranger的结合：** 在Hadoop中，LDAP和Ranger可以结合使用，以实现更全面的用户权限管理和安全控制。
    - 用户身份验证：Hadoop服务可以通过LDAP验证用户的身份，确定用户是否合法。
    - 用户信息管理：LDAP可以作为Ranger的用户存储插件，允许Ranger从LDAP目录中获取用户和组的信息，用于定义访问策略。
    - 统一身份管理：通过LDAP与Ranger的结合，可以实现统一的身份管理和单点登录(SSO)机制，让用户可以使用同一组凭据访问多个应用和服务，提高用户体验。
